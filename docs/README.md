# Автоматизация актуализации документации

Инструмент объединяет в одном интерфейсе два процесса:
1. Подготовка release notes по задачам Jira.
2. Проверка документации на стоп-слова (ban-слова).

Основная цель: сократить ручную рутину при подготовке релизов и сделать формулировки более единообразными.

Подробная спецификация интерфейса: `docs/INTERFACE_SPEC.md`.

## Что умеет инструмент

- Заполнять релизную таблицу в Jira по задачам релиза.
- Генерировать готовый changelog-текст для вставки в исходный код документации.
- Генерировать одну краткую релизную строку по одной задаче.
- Показывать preview релизной таблицы без записи в Jira.
- Сравнивать эталон и сгенерированные формулировки (compare + LLM-судья).
- Проверять очистку чувствительных данных (sanitizer check).
- Сканировать документацию на стоп-слова.

## Быстрый старт

1. Установите Python 3.9+.
2. Установите зависимости:

```bash
pip install -r requirements.txt
```

3. Создайте `.env` на основе `env_example` и заполните его.
4. Запустите UI:

```bash
streamlit run ui/app.py
```

Если команда `streamlit` не найдена:

```bash
python -m streamlit run ui/app.py
```

## Что заполнить в `.env` обязательно

Минимально для работы:

- `JIRA_BASE_URL`
- `JIRA_AUTH_TYPE`
- `JIRA_API_TOKEN`
- `LLM_ENDPOINT`
- `LLM_MODEL`

Если используется `basic`-авторизация в Jira, дополнительно нужен `JIRA_EMAIL`.

Остальные параметры можно оставить как в `env_example` и корректировать по мере необходимости.

## Как получить Jira token

Для Jira Server/DC (ваш текущий сценарий):
1. Откройте профиль Jira.
2. Перейдите в раздел `Персональные токены доступа` (Personal Access Tokens).
3. Создайте новый токен.
4. Скопируйте токен и укажите его в `.env`:
   - `JIRA_API_TOKEN=<ваш_токен>`
5. Проверьте, что `JIRA_BASE_URL` указывает на ваш Jira-инстанс.

## Основной сценарий работы в UI

### 1) Заполнение релизной таблицы в Jira

Вход:
- `release parent` — релизная задача с задачами релиза.
- `release target` — задача, где находится таблица для заполнения.

Результат:
- таблица в `release target` обновляется краткими релизными формулировками.

### 2) Генерация готового changelog-текста

Вход:
- `release target` с уже заполненной релизной таблицей.

Результат:
- готовый текст для вставки в исходный код документации с условными блоками:
  - `{% if domen == 'WCE' %}`
  - `{% else %}`
- можно отредактировать и скопировать прямо из UI.

Дополнительно:
- красный фрагмент в таблице (`{color:#de350b}...{color}`) трактуется как MS-only и автоматически оборачивается в `{% if domen != 'WCE' %}...{% endif %}`.

### 3) Проверка документации на стоп-слова

Вход:
- стартовый URL документации,
- путь к словарю стоп-слов,
- задержка между запросами,
- лимит страниц (по умолчанию `0`, то есть без лимита).

Результат:
- сводка по проверке,
- список проблемных страниц,
- контексты найденных вхождений.

## Дополнительные режимы

В UI также доступны:
- генерация одной релизной строки,
- preview релизной таблицы без записи в Jira,
- compare эталон vs сгенерированное,
- sanitizer check,
- история запусков.

Для CLI-режимов см. `docs/MODES.md`.

## Ограничения

- Режимы, которые изменяют описание Jira-задачи, работают при wiki-формате описания (Jira Server/DC).
- Для Jira Cloud (ADF) может потребоваться отдельная реализация обновления описания.
