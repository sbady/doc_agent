# Task Playbook

Этот файл фиксирует типовые рабочие задачи по проекту документации и порядок действий агента.

Главные сценарии:
- заполнение и редакторская проверка релизной таблицы в Jira
- перенос согласованных релизных заметок из Jira-таблицы в `releases.md`
- создание или изменение статей по Jira-задаче
- подготовка комментариев для Jira после публикации изменений в статьях

Для задач на создание или изменение статей по Jira этот файл является обязательной точкой входа. Агент должен прочитать этот workflow до планирования, выбора статей, создания ветки и финального отчета.

## Общие правила для задач по статьям

- Рабочая ветка в `doc_diplo` должна называться точно так же, как Jira-задача, по которой пользователь поставил работу, например `MSP-9479`
- Если задача расширяет существующий функционал на новый тип мероприятия, роль или контур, нельзя переписывать описание всего функционала как новое частное поведение
- Перед правками нужно определить базовое поведение по текущей статье, связанной Jira-задаче или задаче-источнику, а затем добавлять только изменение из текущей задачи
- Существующие ограничения, роли, контуры и условия доступности нельзя удалять или ослаблять, если текущая Jira-задача явно не говорит, что они изменились
- Если в Jira написано "аналогично задаче MSP-XXXX", "описано в MSP-XXXX", "по аналогии с MSP-XXXX" или дана ссылка на задачу-источник, агент обязан открыть эту задачу до правки статьи
- Если связанную задачу открыть не удалось, нужно явно написать, какой baseline не подтвержден, и не переписывать смежный функционал
- Перед финальным ответом и после создания MR агент всегда проверяет конфликты против всех целевых веток MR. Если есть MR в `DEV` и `main`, проверять нужно оба направления отдельно
- Если `DEV` и `main` существенно разошлись, нельзя вливать `DEV` в ветку для `main`. Нужно подготовить отдельную ветку от `DEV` для DEV-MR и отдельную ветку от `main` для main-MR

## Формат комментария для Jira

После публикации изменений по статье или группе статей комментарий для Jira нужно давать в коротком и естественном формате.

Комментарий для Jira - это отдельный текст для проверяющего, а не Git-отчет. В него нельзя добавлять ссылки на MR, номера коммитов, результат проверки конфликтов, названия веток или технические подробности публикации. Эти данные можно дать пользователю отдельно в финальном отчете, но не внутри блока комментария для Jira.

Комментарий готовится после публикации на дев-стенд. Ссылки в комментарии должны вести именно на дев-стенд документации, а не на prod-страницы из `og:url` и не на локальные файлы. Если публикации на дев-стенд еще не было, агент должен прямо написать, что Jira-комментарий пока нельзя подготовить полностью, потому что нет проверочных dev-ссылок.

Если статья доступна в обоих продуктах, в Jira-комментарии нужно дать dev-ссылки на оба продуктовых домена: MS (`https://diplo-mash.pikemedia.live/ru/...`) и WCE (`https://diplo-wce.pikemedia.live/ru/...`). Если изменение относится только к одному продукту, нужно явно не приписывать его второму продукту; WCE-ссылку давать только для проверки реально изменившихся общих или WCE-блоков.

Базовый шаблон:

```text
Опубликовал изменения. Проверить можно на дев-стенде:

* [Название статьи|https://... ] – кратко, что изменено
* [Название статьи|https://... ] – кратко, что изменено
```

Правила:
- начинать с опубликованных статей, а не с MR или веток
- писать `Проверить можно на дев-стенде`, а не `на dev-стенде`
- между ссылкой и описанием использовать короткое тире `–`, а не дефис `-` и не длинное тире `—`
- писать коротко и по делу, без воды
- перечислять только фактические изменения
- описывать, что именно изменено в статье или скриншотах, чтобы ревьюер мог быстро проверить правку
- для общих статей разделять описание по продуктам, если в MS и WCE изменились разные блоки или скриншоты
- если статья новая, прямо писать, что добавлена новая статья
- если нужно, давать ссылку только на те статьи, где реально были изменения
- формулировки должны быть понятны ревьюеру без погружения в задачу
- не добавлять ссылки на статьи, которые не менялись
- не писать в Jira-комментарии `конфликтов не обнаружено`, `MR создан`, `commit ...` и похожие служебные статусы

## Финальный ответ по задаче на статьи

После завершения задачи агент должен вернуть:
- текущую ветку и подтверждение, что она совпадает с Jira-ключом задачи
- список измененных статей с кратким описанием изменений и указанием задач (`MSP-XXXX`)
- ссылку на merge request в `DEV` (ссылку на MR в `main` — только если запрошена выкатка на прод; по умолчанию main-MR не создаётся)
- результат проверки конфликтов против `DEV` (и `main` — только при прод-выкатке)
- результат проверки на стоп-слова, если правки затрагивали платную функциональность или термины из словаря стоп-слов
- готовый комментарий для Jira в формате выше, если изменения опубликованы на дев-стенд
- если commit, push, MR или публикация на дев-стенд не выполнялись, явно написать, чего именно нет и почему

## Тип задачи: создание или изменение статей по Jira-задаче

### Когда использовать

Используйте этот сценарий, когда пользователь просит:
- создать новую статью по Jira-задаче
- обновить существующую статью по Jira-задаче
- внести правки по задаче из релизной таблицы, где в колонке `Мануал` стоит `ДА`
- проверить, какие статьи нужно менять по задаче
- подготовить ветку и merge request с изменениями статей

### Входные данные

Минимально достаточно Jira-ключа задачи, например `MSP-9479`.

Если пользователь дополнительно дал краткое описание, список связанных задач, стенды, ссылки на статьи или замечания ревьюера, агент должен использовать это как первичный контекст, но не ограничиваться им, если информации недостаточно.

Если контекста мало, агент должен прочитать Jira-задачу по ключу:
- описание задачи
- комментарии
- блок `Документация Diplodoc`
- указанные стенды
- связанные задачи, если из описания видно, что основная задача агрегирует несколько продуктовых задач

Если Jira недоступна из текущей среды, агент должен явно сказать, какой информации не хватает, и работать только с тем контекстом, который дал пользователь.

### Планирование внутри задачи

Для одной Jira-задачи обычно не нужен отдельный агент-планировщик и отдельный агент-исполнитель. Лучше, чтобы один агент:
1. Собрал контекст.
2. Составил короткий план изменений.
3. Нашел статьи и проверил оглавление.
4. Сразу внес изменения.
5. Проверил diff и подготовил результат.

Разделять на разных агентов имеет смысл только если одновременно выполняется много независимых Jira-задач. В таком случае безопасный формат - один агент на одну Jira-задачу, чтобы не раздувать контекст и не смешивать изменения разных задач.

### Как определить статьи

Агент должен искать целевые статьи в `doc_diplo/src/ru/documentation`.

Обязательно проверить оглавление:
- `doc_diplo/src/ru/toc.yaml` - русская документация
- `doc_diplo/src/en/toc.yaml` - английская документация, если задача затрагивает EN

Оглавление нужно использовать, чтобы:
- понять структуру разделов
- найти ближайшую существующую статью
- проверить, нужно ли добавлять новую статью в навигацию
- не создать дубль статьи, если подходящая уже есть

**Следовать ссылкам между статьями (обязательно).** Если целевая или связанная статья ссылается на другую статью, нужно открыть связанную и проверить, не стала ли там информация неактуальной из-за текущей правки — и при необходимости актуализировать её в этой же задаче. Продуктовые решения (например переименование функции) выносятся на согласование, не выдумываются.

**Сначала понять, как работает функционал.** До решения о правках установить фактическое поведение: пройти цепочку связанных Jira-задач (связанные задачи и задачи, на которые ссылаются они) и собрать все связанные статьи через полнотекстовый поиск (`rg`) и `toc.yaml`. Только после этого решать, что именно и где актуализировать.

Если задача относится только к WCE или только к MS, агент должен проверить условные блоки `{% if domen == 'WCE' %}` / `{% else %}` и не допустить утечки контурной информации в другой стенд.

Особое внимание — **платной функциональности**: продажа билетов, заказы, оплата, скидки, промокоды и т.п. есть в продукте MS и **отсутствуют в WCE**. Такие упоминания нельзя оставлять в общем тексте или в WCE-ветке условного блока. Если правки затронули платность или другие термины из словаря стоп-слов, после публикации на дев-стенд нужно прогнать сканер стоп-слов (`stop_words_checker-main/`, см. `Project Docs/04-doc-agent-reference.md`) по WCE-стенду и приложить результат. Это часть Definition of Done в `AGENTS.md`.

### Базовый процесс

1. Проверить состояние `doc_diplo` и текущую ветку.
2. Создать или переключиться на рабочую ветку по ключу поставленной Jira-задачи, если пользователь просит вносить изменения. Имя ветки должно точно совпадать с ключом задачи.
3. Прочитать Jira-задачу и связанные задачи, если они важны для смысла.
4. Если текущая задача ссылается на задачу-источник или формулировку "аналогично", прочитать связанную задачу до правок.
5. Определить baseline существующего функционала: где он уже описан, для каких типов мероприятий, ролей и контуров доступен, какие ограничения уже действуют.
6. Определить стенды и аудиторию статьи: организатор, администратор, участник, модератор.
7. Найти существующие статьи через `rg` и `toc.yaml`.
8. Решить, нужна новая статья или достаточно обновить существующую.
9. Перед большими правками кратко сформулировать план: какие статьи меняются и почему.
10. Внести только delta текущей задачи: что добавилось или изменилось относительно baseline.
11. Сохранить стиль Diplodoc/YFM: существующие условные блоки, `cut`, `note`, ссылки, якоря и формат изображений.
12. Проверить diff: не должно быть лишних файлов, мусора из других задач и случайных правок.
13. По умолчанию (если пользователь не сказал "только черновик" / "не пушь"): сделать commit и push рабочей ветки и **опубликовать в `DEV`**. В сообщении коммита указать, **какие статьи изменены и в рамках каких задач** (`MSP-XXXX`); авторство инструментов/моделей (`Co-Authored-By`, Claude, Codex) не указывать.
14. По умолчанию создать **только MR в `DEV`** и проверить конфликты против `DEV`. **MR в `main` по умолчанию НЕ создавать** — он висел бы открытым весь цикл согласования и правок. Выкатка в `main` (ветка от `main` + MR в `main`) — отдельный шаг, который инициирует пользователь, когда задача согласована.
15. Вернуть короткий отчет: ветку(и), изменённые статьи с задачами, ссылку на MR в `DEV` (на `main` — только если запрошена прод-выкатка), результат проверки конфликтов и готовый комментарий для Jira, если есть dev-ссылки. Если шаг не выполнялся — явно написать, чего нет и почему.

### Правила содержания статьи

Статья должна объяснять пользовательский сценарий, а не пересказывать Jira.

Статья описывает **текущий факт**: что можно делать в интерфейсе и как — а не то, что «недавно изменилось». В теле статьи **не используются формулировки изменения во времени** — «теперь», «уже», «стало доступно», «появилась возможность». Фиксируется факт: не «теперь доступна печать бейджа», а «доступна печать бейджа»; не «стало доступно только администратору», а «доступно только администратору». (Формулировки с «теперь» уместны в релизных заметках, где смысл именно в том, чтобы показать изменение, — но не в статье.)

Писать нужно:
- где находится настройка или действие в интерфейсе
- кому доступен функционал
- что изменилось для пользователя
- как выполнить действие
- какие ограничения есть в интерфейсе
- где посмотреть результат, статистику или выгрузку, если это часть сценария

Не писать без необходимости:
- внутренние названия сервисов
- backend/frontend-детали
- API, endpoint, таблицы БД, технический долг
- историю разработки
- формулировки из Jira, которые полезны разработке, но не помогают пользователю

### Работа с несколькими задачами внутри одной Jira-задачи

Если задача на документацию агрегирует несколько продуктовых задач, агент должен обработать каждую продуктовую задачу отдельно:
- определить, какая статья относится к этой продуктовой задаче
- проверить, не нужна ли новая статья
- не смешивать несвязанные изменения в одном абзаце
- в финальном комментарии перечислить изменения по статьям, а не по внутренним подзадачам

Для `MSP-9479` пример такого разбиения:
- `MSP-9368` - красивая ссылка на лендинг для мероприятий со свободным входом, вероятная статья: `Лендинг`
- `MSP-8547` - асинхронная выгрузка пользователей портала, вероятная статья: раздел портала с отчетами

Если продуктовая задача говорит, что работа блока аналогична другой задаче, эта другая задача становится обязательным источником baseline. Например, для `MSP-9368` нужно сначала прочитать `MSP-6362`, потому что текущая задача только расширяет существующую логику ссылки на мероприятия со свободным входом.

### Если информации недостаточно

Агент не должен выдумывать поведение интерфейса.

Если после чтения Jira и статей остаются вопросы, нужно остановиться и коротко перечислить:
- какая статья вероятно нужна
- какой фрагмент поведения непонятен
- какой вопрос нужно уточнить у пользователя или аналитика

Если можно безопасно внести часть изменений, агент может сделать это и отдельно отметить оставшиеся вопросы.

## Тип задачи: заполнение и проверка релизной таблицы

### Цель процесса

Работа с документацией релиза привязана к релизу продукта. Процесс включает:
- создание релизных заметок - кратких описаний задач релиза
- определение задач, для которых нужно изменить существующие статьи или создать новые
- подготовку таблицы к согласованию с ответственным менеджером или техписом

Итоговая таблица должна быть не просто автоматически заполненной, а доведенной до состояния, в котором техпису остается финально проверить ее и передать на согласование.

### Когда использовать

Используйте этот сценарий, когда пользователь просит:
- заполнить релизную таблицу
- проверить уже заполненную черновую релизную таблицу
- отсортировать релизную таблицу
- поправить формулировки в релизной таблице
- отметить задачи, которые требуют изменений в документации

### Входные данные

Для заполнения таблицы нужны два ключа Jira:
- `release_parent` - задача релиза, из которой нужно взять связанные задачи
- `release_target` - задача вида `✒️ Актуализация документации релиза`, куда нужно записать таблицу

Пример пользовательской команды:

```text
Заполни релизную таблицу:
release_parent = MSP-9419
release_target = MSP-9453
```

Также допустима естественная формулировка:

```text
Таблицу нужно заполнить в MSP-9453, задачи релиза брать из MSP-9419
```

Агент должен понимать это так:
- `MSP-9419` - источник задач релиза
- `MSP-9453` - целевая задача, где находится релизная таблица

### Базовый процесс для техписа

1. Создается задача актуализации документации релиза по шаблону `✒️ Актуализация документации релиза`.
2. В шаблоне задачи уже есть таблица релиза.
3. В таблицу добавляются все задачи релиза и краткое описание того, что было изменено.
4. В колонке `Мануал` отмечаются задачи, которые требуют изменения существующих статей или создания новых.
5. Таблица проверяется, сортируется и согласуется.
6. По задачам с отметкой в `Мануал` создаются отдельные задачи на создание или изменение статей.
7. После согласования таблица переносится в документацию в раздел `Релизы`.

## Основной сценарий агента: заполнить черновик и довести таблицу

### 1. Проверить готовность `doc_agent`

Перед запуском нужно убедиться, что:
- существует `doc_agent\.venv`
- в `doc_agent\.env` настроены Jira и LLM
- доступны значения:
  - `JIRA_BASE_URL`
  - `JIRA_API_TOKEN`
  - `JIRA_AUTH_TYPE`
  - `LLM_ENDPOINT`
  - `LLM_API_KEY`
  - `LLM_MODEL`

### 2. Заполнить черновую таблицу

По умолчанию агент может сразу заполнять таблицу в Jira через `doc_agent`, не делая обязательный `fill-preview`.

Команду запускать из директории `doc_agent`:

```powershell
& ".\.venv\Scripts\python.exe" ".\main.py" --release-parent MSP-9419 --release-target MSP-9453 --mode fill
```

Что делает `fill`:
- берет связанные задачи из `release_parent`
- генерирует краткие описания
- вставляет или обновляет блок релизной таблицы в `release_target`
- сохраняет уже существующие значения в колонках `Шаблоны` и `Мануал`

`fill-preview` использовать, если пользователь явно просит сначала посмотреть черновик без записи или если есть риск массово испортить уже вручную вычитанную таблицу.

### 3. Проверить полноту таблицы

После заполнения агент должен проверить:
- все ли связанные задачи релиза попали в таблицу
- нет ли лишних строк, не относящихся к релизу
- не попали ли подзадачи, если они не должны описываться отдельно
- сохранены ли ручные значения в колонках `Шаблоны` и `Мануал`
- нет ли пустых строк или сломанного Jira wiki-формата

Если таблица собирается из нескольких релизных задач, агент должен:
- собрать полный список связанных задач из каждого `release_parent`
- исключить саму целевую задачу актуализации документации, если она связана с релизами
- дедублицировать задачи, если одна задача связана с несколькими релизами
- сверить финальную таблицу с полным списком связанных задач, а не только с результатом `fill-preview`

`fill-preview` может не показать часть связанных задач, например технические подзадачи. Такие задачи нельзя молча терять: их нужно добавить в таблицу и классифицировать. Если это техническая подзадача без пользовательского эффекта, она остается в таблице с `не описываем` и переносится в технический хвост.

### 4. Отсортировать таблицу

Таблица должна иметь стабильную структуру.

Порядок групп:
- `MS` - `Фича`
- `MS` - `Баг`
- `Оба` и `Оба?` - `Фича`
- `Оба` и `Оба?` - `Баг`
- `WCE` - общие задачи - `Фича`
- `WCE` - общие задачи - `Баг`
- `WCE` - работы над порталом `pro.t1.ru`
- `WCE` - работы над порталом `Импульс`
- `WCE` - работы над конструктором форм
- технические и внутренние задачи с `не описываем`

Внутри каждой группы задачи сортируются по ключу Jira.

Если одна таблица объединяет несколько релизов, агент не должен пытаться разделять ее на несколько таблиц без отдельной команды пользователя. В таком случае все строки сортируются как единый список по общим правилам.

Технические задачи не удаляются из таблицы. Если задачу не нужно публиковать в релизных заметках, в колонке `Краткое описание` нужно написать `не описываем` и перенести строку в конец таблицы.

### Правила специальных WCE-групп

Специальные WCE-группы определяются только по эмодзи-маркерам в названии задачи:
- `🩲` -> `Работы над порталом pro.t1.ru`
- `👙` -> `Работы над порталом Импульс`
- `👽` -> `Работы над конструктором форм`

Дополнительные правила:
- если у задачи нет одного из этих маркеров, она остается в общей WCE-группе
- если в названии несколько разных специальных эмодзи из списка `🩲`, `👙`, `👽`, задача остается в общей WCE-группе
- если рядом со специальным эмодзи есть нерouting-эмодзи, например `🔬👽`, это не отменяет специальную группу
- агент не должен определять специальную группу по смыслу задачи, только по формальному маркеру
- эмодзи `🦦` не является формальным маркером специальной группы
- если в задаче есть `🦦`, это может быть косвенным признаком WCE, но не основанием относить задачу к `Импульсу`, `pro.t1.ru` или формам

Эти правила нужны уже на этапе проверки релизной таблицы, а не только при переносе релиза в `releases.md`.

### 5. Проверить и поправить колонку `Стенд`

Агент должен проверить стенд по описанию задачи и комментариям.

Ожидаемые значения:
- `MS`
- `WCE`
- `Оба`
- `Оба?`

`Оба?` допустимо только если стенд не удалось определить уверенно. Если стенд можно определить по задаче, `Оба?` нужно заменить на конкретное значение.

### 6. Проверить и поправить колонку `Тип задачи`

Ожидаемые значения:
- `Фича`
- `Баг`

Тип задачи не нужно менять механически только потому, что в описании есть слова про ошибку, сбой или некорректный статус.

Если задача заведена как фича или техническое улучшение и ее можно описать через положительный пользовательский эффект, она может оставаться `Фича`. Например: улучшена стабильность загрузки, статус обновляется корректно, сообщения стали понятнее.

`Баг` ставится, если задача явно является пользовательской неисправностью: некорректное отображение, ошибка поведения, недоступная кнопка, сбой в пользовательском сценарии. Если есть сомнение, агент не должен менять тип без явного основания из Jira-задачи или без согласования.

### 7. Проверить и поправить `Краткое описание`

Хорошая строка:
- описывает пользовательский результат, а не внутреннюю реализацию
- формулируется через положительный эффект для пользователя, а не через самокритику продукта
- короткая, но достаточно конкретная
- понятна без чтения Jira-задачи
- использует терминологию интерфейса продукта
- не обещает больше, чем реально сделано
- не уходит в технические детали: сервисы, таблицы БД, SQL, API, endpoint, panic, DNS, логи, внутренние коды ошибок

Термины для релизных формулировок:
- не писать `админка`; писать `административный интерфейс`
- не писать `чипсы`; писать `информационные метки`, если речь о визуальных метках на лендинге или в интерфейсе
- не писать `фавиконка` или `favicon`; писать `значок вкладки браузера`
- не использовать внутренние названия блоков без контекста: если упоминается `счет`, уточнить, что это счет на оплату участия или счет в заказе

Частицу `теперь` не нужно механически убирать из релизных формулировок. Она полезна, когда нужно явно показать изменение поведения после релиза: раньше логика работала иначе или функциональность была недоступна, а после релиза работает по-новому. Формулировки вида `теперь ...`, `теперь можно ...`, `теперь корректно ...` допустимы и часто предпочтительны, если они помогают показать, что именно изменилось. При этом не нужно добавлять `теперь` в каждую строку, если изменение и так понятно из контекста. Это правило касается **только релизных формулировок** (таблица и релизные заметки); в теле статьи изменение во времени не описывается — см. «Правила содержания статьи».

Даже для багов не нужно без необходимости писать так, будто продукт был в плохом состоянии. Предпочитайте формулировки вида `теперь ... корректно`, `улучшено ...`, `стало стабильнее ...`, если они точно передают суть изменения.

Если формулировка сомнительная, агент должен провалиться в исходную Jira-задачу и свериться:
- с описанием
- с комментариями
- с указанными стендами
- с фактическим пользовательским эффектом

Агент может править формулировку сам, если смысл задачи понятен и правка не требует продуктового решения.

Если смысл задачи неясен или есть риск изменить значение, агент должен подсветить строку пользователю, а не выдумывать.

### 8. Когда писать `не описываем`

`не описываем` уместно, если задача:
- внутренний техдолг без пользовательского эффекта
- инфраструктурная настройка без заметного изменения интерфейса или поведения
- исследование, диагностика или рефакторинг
- внутренняя стабилизация без отдельного пользовательского сценария
- содержит эмодзи `🥖` или `🔌` в названии задачи

Если задача исправляет видимую ошибку, поведение интерфейса, письмо, отображение, права, доступность кнопки или пользовательский сценарий, ее нужно описывать.

Если задача выглядит технической, но в ней есть явный пользовательский эффект в интерфейсе, ее не нужно автоматически переносить в `не описываем`. Например: отображение понятной ошибки при загрузке файла, корректное обновление статуса загрузки, исправление сбоя в административном интерфейсе.

Если краткое описание уже было испорчено санитайзером, агент должен восстановить безопасную пользовательскую формулировку. Например, диапазон ширины экрана `1240-1350` не должен превращаться в `<redacted_phone>`.

### 9. Заполнить колонку `Мануал`

Колонка `Мануал` нужна, чтобы отметить задачи, по которым требуется создание или изменение статей документации.

Главный источник для первичной проверки - блок `Документация Diplodoc` в Jira-задаче.

В задачах этот блок часто содержит один из вариантов:
- `требуется новая статья`
- `требуется обновить существующую статью`
- `не требуется`

Если в блоке явно оставлено `требуется новая статья` или `требуется обновить существующую статью`, в колонке `Мануал` нужно ставить `ДА`, но после этого все равно проверить, какая статья реально нужна.

Если в блоке явно оставлено `не требуется`, в колонке `Мануал` обычно остается `-`, если агент не видит явного пользовательского сценария, который точно требует документации.

Если в блоке осталась дефолтная строка с несколькими вариантами через `/`, например `требуется новая статья`/`требуется обновить существующую статью`/`не требуется`, это не выбранное значение. Агент не должен автоматически считать такую строку требованием к документации. Такую задачу нужно подсветить в отчете как требующую ручной проверки.

Ставить `ДА`, если задача:
- добавляет новый пользовательский функционал
- меняет существующий пользовательский сценарий
- меняет название, поведение или доступность элементов интерфейса
- меняет логику настроек, прав, уведомлений, писем, статистики, отчетов или выгрузок
- требует новой статьи
- требует актуализации существующей статьи

Ставить `-`, если:
- задача не требует изменений в пользовательской документации
- задача отмечена как `не описываем`
- изменение внутреннее и не влияет на статьи

Если в колонке `Мануал` уже есть ручное значение, агент не должен затирать его без причины. Если значение кажется ошибочным, агент должен подсветить это в отчете и предложить правку.

### 10. Использовать режимы проверки `doc_agent`

Для уже заполненной таблицы доступны:

```powershell
& ".\.venv\Scripts\python.exe" ".\main.py" --release-target MSP-9453 --mode refine-preview
```

`refine-preview` показывает варианты стилистического улучшения формулировок без записи.

```powershell
& ".\.venv\Scripts\python.exe" ".\main.py" --release-target MSP-9453 --mode refine-apply
```

`refine-apply` применяет улучшения к колонке `Краткое описание`.

Важно:
- `refine-preview` и `refine-apply` не сортируют таблицу
- `refine-apply` сохраняет текущий порядок строк
- эти режимы не заменяют редакторскую проверку агентом
- эти режимы не переносят технические строки в конец и не принимают решение о `не описываем`

Для сравнения ручных и сгенерированных формулировок можно использовать:

```powershell
& ".\.venv\Scripts\python.exe" ".\main.py" --release-target MSP-9453 --mode compare --compare-mode replace
```

`compare` полезен, если в таблице уже есть ручные формулировки и нужно оценить, где автоматическая генерация расходится с ними.

### 11. Финальный отчет пользователю

После доведения таблицы агент должен сообщить:
- какая Jira-задача обновлена
- сколько строк проверено
- какие строки изменены
- какие строки оставлены как `не описываем`
- какие задачи отмечены в `Мануал`
- какие строки требуют ручного решения, если такие остались

## Тип задачи: финальная вычитка уже заполненной таблицы

### Когда использовать

Используйте этот сценарий, когда пользователь говорит:
- таблица уже заполнена
- нужно ее проверить
- нужно отсортировать
- нужно довести формулировки до публикационного качества
- нужно проставить `Мануал`

### Что делает агент

1. Читает релизную таблицу из `release_target`.
2. Сверяет задачи таблицы со связанными задачами `release_parent`, если он указан.
3. Проверяет порядок строк и приводит его к правилам сортировки.
4. Проверяет `Стенд`, `Тип задачи`, `Краткое описание`, `Мануал`.
5. По спорным строкам открывает исходные Jira-задачи.
6. Правит очевидные ошибки.
7. Не меняет спорные строки без уверенности.
8. Возвращает короткий отчет о правках и оставшихся вопросах.

### Важное отличие от согласованной таблицы

Черновая таблица может редактироваться агентом.

Согласованная таблица считается read-only. Если пользователь прямо сказал, что таблица согласована, агент не должен менять ее без отдельного разрешения.

## Тип задачи: перенос согласованных релизных заметок в документацию

### Когда использовать

Используйте этот сценарий, когда:
- релизная таблица в Jira уже заполнена
- таблица согласована вручную
- нужно перенести итоговые релизные заметки в `releases.md`

### Входные данные

Для выполнения задачи нужны:
- `release_target` - Jira-задача, где лежит согласованная релизная таблица
- `release_version` - номер релиза, который нужно добавить в документацию

Пример:
- `release_target = MSP-9453`
- `release_version = 6.78.1`

### Жесткое ограничение

После согласования таблицы агент:
- не должен сам менять релизную таблицу в Jira
- не должен автоматически править формулировки в согласованной таблице
- не должен локально перефразировать или улучшать формулировки при переносе в `releases.md`
- может только подсветить замечания, если замечает спорные или явно странные строки

Если при переносе обнаружена проблема:
- агент останавливается
- коротко описывает замечание
- ждет отдельного согласования на изменение Jira-таблицы

### Получить changelog preview через `doc_agent`

Команду запускать из директории `doc_agent`:

```powershell
$env:PYTHONIOENCODING='utf-8'
& ".\.venv\Scripts\python.exe" ".\main.py" --release-target MSP-9453 --mode changelog-preview
```

Что делает режим:
- читает согласованную таблицу из `release_target`
- собирает готовый текст для `releases.md`
- учитывает условные блоки для `WCE` и `MS`
- пропускает строки `не описываем`
- формирует специальные WCE-блоки по эмодзи-маркерам

Агент должен использовать этот режим, а не собирать релизный текст вручную.

### Проверить changelog preview

Перед переносом в документацию нужно проверить:
- текст действительно собран из согласованной таблицы
- в релиз не попали строки `не описываем`
- не сломались условные блоки для `WCE` и `MS`
- нет явных дублей
- нет пустых секций
- формулировки совпадают с колонкой `Краткое описание`
- специальные WCE-блоки сформированы только по формальным эмодзи-маркерам

### Обновить `releases.md`

Нужно:
- создать ветку от `main`
- создать ветку от `DEV`
- вставить новый блок релиза в `doc_diplo/src/ru/documentation/releases.md`
- соблюдать структуру страницы релизов:
  - под заголовком `# Релизы платформы` должен быть только последний релиз
  - заголовок последнего релиза указывается без даты, например `## Релиз 6.79`
  - блок `## Предыдущие релизы` должен идти сразу после последнего релиза
  - предыдущий открытый релиз нужно перенести в `## Предыдущие релизы` первым `cut`-блоком
  - дата релиза указывается только в названии `cut`-блока предыдущего релиза, например `{% cut "_Релиз 6.78.1 от 18.05.2026_" %}`
  - одновременно не должно быть двух открытых релизов перед блоком `## Предыдущие релизы`
- сохранить форматирование файла
- не задеть старые релизы
- проверить diff
- сделать commit и push в обе ветки
- вернуть ссылки на MR в `main` и `DEV`

### Минимальный чек-лист перед завершением

Перед тем как считать перенос выполненным, проверить:
- таблица действительно согласована
- changelog preview собран из нужной Jira-задачи
- Jira-таблица не изменялась без разрешения
- новый релиз добавлен в `releases.md`
- старые релизы не повреждены
- изменен только ожидаемый файл
- ветки для `main` и `DEV` запушены
- ссылки на MR отданы пользователю
