# Техническая спецификация и логика проекта
python main.py --mode fill
Цель документа — зафиксировать текущие возможности и правила работы инструмента, чтобы новые фичи развивались последовательно и не противоречили существующей архитектуре. Документ поддерживается актуальным и служит опорой для планирования.

## Область применения
- Генерация краткого релизного резюме по одной задаче Jira (single-issue summary).
- Автоматизация формирования и обновления таблицы релиз‑нот по связанным задачам родительской релизной задачи (release notes table).

## Термины
- Родительская релизная задача — задача, с которой связаны задачи релиза через связь Relates («связаны с»).
- Целевая задача — задача, в чьём описании хранится таблица релиза.
- Стенд — MS/WCE/Оба/Оба? согласно описанию задачи.

## Архитектура и модули
- `config.py` — загрузка конфигурации из `.env`, валидация путей и значений.
- `jira_client.py` — HTTP‑доступ к Jira (чтение задач, обновление описания), извлечение полей.
- `llm_client.py` — вызов локального OpenAI‑совместимого LLM, рендеринг промптов (Jinja2), извлечение ответа.
- `release_notes.py`
  - Сбор связанных задач `Relates` от родителя (без подзадач).
  - Классификация стенда и типа задачи, генерация краткого описания через LLM.
  - Сортировка и построение таблицы (Jira wiki)
  - Идемпотентное обновление существующей таблицы с сохранением колонок «Шаблоны»/«Мануал».
- `prompt_templates/`
  - `release_summary.txt` — общий шаблон резюме релиза (single-issue).
  - `issue_short_summary_feature.txt` — шаблон краткого описания для строки таблицы (Фича).
  - `issue_short_summary_bug.txt` — шаблон краткого описания для строки таблицы (Баг).
  - `issue_short_summary.txt` — опциональный общий шаблон (если нужен единый промпт для обоих типов).

## Настройки окружения (.env)
- Jira
  - `JIRA_BASE_URL` — базовый URL Jira.
  - `JIRA_AUTH_TYPE` — `basic` или `pat`.
  - `JIRA_EMAIL` — email (для `basic`, для `pat` опционально).
  - `JIRA_API_TOKEN` — пароль/API token или PAT.
  - `JIRA_API_VERSION` — `3`/`latest` и т.п.
- Быстрый запуск одиночного резюме
  - `JIRA_ISSUE_KEY` — используется только в одиночном режиме, когда релиз‑режим не активирован.
- LLM
  - `LLM_ENDPOINT` — URL эндпоинта (поддержан `/v1/chat/completions`).
  - `LLM_API_KEY` — опционально.
  - `LLM_MODEL` — обязательная модель для chat‑эндпоинта.
  - `LLM_TEMPERATURE`, `LLM_MAX_TOKENS` — основные параметры.
- Шаблоны
  - `PROMPT_TEMPLATE_PATH` — путь к шаблону общего резюме.
  - `ISSUE_SHORT_FEATURE_TEMPLATE_PATH` — путь к шаблону краткого описания для фич.
  - `ISSUE_SHORT_BUG_TEMPLATE_PATH` — путь к шаблону краткого описания для багов.
  - `ISSUE_SHORT_TEMPLATE_PATH` — опционально: общий оверрайд‑шаблон для обоих типов.
  - `ISSUE_SHORT_MAX_TOKENS` — лимит токенов для краткого описания.
- Технические
  - `REQUEST_TIMEOUT` — таймаут запросов в секундах.
  - `LOG_LEVEL` — уровень логирования (`DEBUG`/`INFO`/`WARNING`/`ERROR`).
- Релиз‑режим
  - `RELEASE_PARENT_KEY` — родительская задача релиза. Если задан, активирует релиз‑режим.
  - `TARGET_ISSUE_KEY` — целевая задача для режима `fill`.

Приоритеты и отсутствие конфликтов
- Если задан `RELEASE_PARENT_KEY` или передан `--release-parent`, запускается релиз‑режим и `JIRA_ISSUE_KEY` игнорируется (это ожидаемое поведение, конфликта имён нет).
- В одиночном режиме `JIRA_ISSUE_KEY` используется как дефолт, если позиционный аргумент не передан.

## CLI: режимы работы
- Одиночный режим (старый поток)
  - `python main.py MSP-1234 [--json] [--log-level DEBUG]`
  - Или без ключа — берётся `JIRA_ISSUE_KEY`.
- Релиз‑режим
  - Просмотр: `python main.py --release-parent MSP-7288 --mode view`
  - Заполнение: `python main.py --release-parent MSP-7288 --release-target MSP-7771 --mode fill`

## Алгоритмы и правила
1) Сбор задач
- Из родителя берутся связанные задачи типа Relates (оба направления), исключая самого родителя.
- Подзадачи пропускаются.

2) Классификация стенда (по описанию)
- Если содержит «Mashroom RU» → `MS`.
- Если содержит «We.Cloud» → `WCE`.
- Если оба значения → `Оба`.
- Если поле не обнаружено или нет узнаваемого значения → `Оба?`.

3) Тип задачи
- `Bug` → `Баг`.
- Остальные типы → `Фича`.

4) Краткое описание
- Генерируется отдельно для каждой задачи через LLM по шаблону, выбранному по типу задачи (`Фича`/`Баг`).
- Если задача сугубо техническая — LLM должен вернуть «не описываем».
- Запросы к LLM выполняются последовательно (без параллелизма).

5) Сортировка
- Группы: MS → Оба/Оба? → WCE.
- Внутри группы: Фичи сверху, затем Баги; далее по ключу задачи.

6) Таблица (Jira wiki)
- Заголовок: `h2. Таблица релиза: в какие статьи нужно внести изменения`.
- Шапка: `||Задача||Стенд||Тип задачи||Шаблоны||Мануал||Краткое описание||`.
- Строка: `|[KEY Название|URL]|MS/WCE/Оба/Оба?|Баг/Фича|-|-|краткое описание|`.

7) Обновление таблицы (идемпотентность)
- Ищется существующая секция по заголовку h2.
- Если таблица есть — построчное обновление: строки задач обновляются, новые добавляются; колонки «Шаблоны» и «Мануал» сохраняются.
- Если таблицы нет — создаётся новая секция и добавляется в описание.

## Логирование и аудит
- INFO: подключение к Jira, краткая сводка задач, отправка/получение LLM, обновление целевой задачи.
- DEBUG: сырой JSON от Jira, превью промптов/ответов, таблица перед обновлением.
- Ошибки: HTTP‑коды и тело ответа, таймауты, невозможность обновления (ADF и т.п.).

## Ограничения
- Автозаполнение описания поддержано для Jira с wiki‑строкой (Server/DC). Для Jira Cloud (ADF) — требуется отдельная реализация ADF.
- Поле «Стенды» извлекается эвристически из описания, отсутствует — ставится `Оба?`.

## Производительность
- Ожидаемо ~30 задач. Параллельные вызовы LLM не используются, чтобы не деградировать локальную модель.

## Безопасность
- Чтение исходных задач — только чтение.
- Перед записью в целевую задачу выполняется запрос на чтение описания; при ошибке запись не выполняется.
- Секреты не логируются.

## Планы и контроль изменений
