# Чувствительные данные, подлежащие маскировке

## Креды и токены
- HTTP-заголовки авторизации:
  - `Authorization: Basic ...` → маскировать полностью.
  - `Authorization: Bearer ...` → маскировать полностью.
- Строки/фрагменты с ключевыми словами (без учёта регистра):
  - `token`, `secret`, `password`, `passwd`, `pass`, `key` (вида `key=...`, `key: ...`).
  - Префиксы `sk-...`, длинные base64/hex-подобные строки (≥ 20 символов).
- Персональные токены/секреты в curl/скриптах даже без слова Authorization.

## PII / контакты
- E-mail адреса.
- Телефоны.
- IP адреса.

## URL
- Сохраняем домен и путь, удаляем query/fragment (все параметры после `?` и `#`).
- Инлайновые вложения Jira `!image...!` — удаляем полностью.

## Форматы, которые часто содержат секреты
- Любые base64/hex-подобные длинные токены (например, JWT, ключи доступа).
- Connection strings (если встречаются): базы, брокеры, хранилища. (В текущем санитайзере явно не выделены, но длинные токены/пароли внутри будут замаскированы.)
- Cookies/Set-Cookie (если встречаются в логах) — могут содержать токены сессий, стоит маскировать при появлении.

## Что НЕ режем
- Шаги воспроизведения, текстовое описание багов/фич.
- Кодовые блоки целиком (не удаляем), но внутри них маскируем секреты и чувствительные куски по указанным правилам.
- Домены и пути URL — оставляем для контекста.

## Что маскируется сейчас (в коде)
- Authorization: Basic/Bearer → `<redacted>`.
- `token|secret|password|passwd|pass|key` с присваиванием → `<redacted>`.
- `sk-...`, длинные base64/hex-подобные строки → `<redacted>`.
- E-mail → `<redacted_email>`, телефон → `<redacted_phone>`, IP → `<redacted_ip>`.
- URL — без query/fragment.
- Inline Jira attachments `!image...!` — удаляются.
- Обрезка сверхдлинных текстов (6000+ символов) с припиской `<truncated>`.
