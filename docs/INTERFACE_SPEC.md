# Спецификация интерфейса: Release Notes + Doc QA

## 1. Цель
Сделать единый интерфейс для двух рабочих процессов:
1. Актуализация документации релиза.
2. Проверка документации на стоп-слова.

Интерфейс должен быть понятен пользователю, который запускает инструмент впервые: с пояснениями, предупреждениями и прозрачными логами.

## 2. Бизнес-процесс (фиксированная версия)

## 2.1 Актуализация документации релиза
1. На выходе релиза создается задача по шаблону `✒️ Актуализация документации релиза`.
2. В этой задаче используется таблица релиза.
3. Для задач релиза формируются краткие описания (релизные заметки).
4. В таблице отмечаются задачи, требующие обновления/создания статей (`Мануал`).
5. Таблица согласуется с ответственными лицами.
6. По отмеченным пунктам создаются задачи `✒️ Создание/изменение статьи`.

## 2.2 Создание/изменение статей
1. Создаются задачи на изменение/создание статей на основе согласованной релизной таблицы.
2. Статьи публикуются на dev для ревью.
3. После правок и согласования публикуются на PROD.

## 3. Информационная архитектура интерфейса

## Раздел A. Обзор процесса
Назначение: onboarding и общая навигация по процессу.

Содержимое:
- краткая схема шагов (релиз -> таблица -> статьи -> проверка -> публикация);
- расширяемый блок с подробным описанием процесса;
- расширяемый блок с промптом для изменения статьи по Jira-задаче;
- краткая инструкция, как использовать этот промпт.

## Раздел B. Релизные заметки
Все функции независимы, без мастер-потока.

### B1. Заполнение релизной таблицы в Jira
- Вход:
- `release parent` (релизная задача с привязанными задачами релиза);
- `release target` (задача с таблицей, куда пишем результат).
- Выход: обновленная таблица в Jira.
- Команда: `python main.py --release-parent ... --release-target ... --mode fill`
- Обязательное предупреждение: target-задача будет изменена.

### B2. Генерация текста для вставки в исходный код документации
- Вход:
- `release target` (задача с уже заполненной релизной таблицей).
- Выход:
- готовый changelog-текст в формате с блоками `{% if domen == 'WCE' %}`/`{% else %}` для вставки в исходный код документации;
- возможность редактирования и копирования в буфер обмена.
- Команда: `python main.py --release-target ... --mode changelog-preview`
- Логика:
- парсится релизная таблица в `release target` (порядок строк сохраняется);
- строки раскладываются по блокам `общие`, `WCE`, `MS`;
- текст, выделенный красным через `{color:#de350b}...{color}`, считается *MS-only* и автоматически оборачивается в `{% if domen != 'WCE' %}...{% endif %}`.

### B3. Одна релизная строка
- Вход: ключ Jira-задачи.
- Выход: краткое описание + возможность скопировать.
- Команда: `python main.py --issue-short MSP-1234`

### B4. Preview таблицы без записи в Jira
- Вход: `release parent`.
- Выход: таблица в интерфейсе (не файл), с возможностью редактирования и копирования.
- Команда (под капотом): `python main.py --release-parent ... --mode view`

### B5. Compare эталон vs сгенерированное
- Вход:
- `release parent`;
- `release target`;
- режим `replace|append`;
- опционально `skip judge`.
- Выход:
- target-таблица дополняется блоком `_Сгенерировано:_`;
- LLM-оценка публикуется комментарием в Jira.
- Команда: `python main.py --release-parent ... --release-target ... --mode compare --compare-mode ...`
- Обязательное предупреждение: таблица в target изменяется.

### B6. Проверка санитайзера
- Вход: ключ Jira-задачи.
- Выход: отчет raw vs sanitized.
- Команда: `python scripts/check_sanitizer.py MSP-8440`

Примечание: `Refine` намеренно исключен из интерфейса MVP.

## Раздел C. Проверка документации на стоп-слова (Doc QA)

### C1. Сканирование
- Вход:
- стартовый URL документации;
- путь к словарю стоп-слов;
- задержка между запросами;
- лимит страниц (опционально).
- Выход:
- summary (страниц проверено, вхождений, проблемных страниц);
- список проблемных страниц;
- контексты совпадений.

### C2. Профили словарей (post-MVP)
- несколько словарей для разных продуктов;
- редактирование словаря из интерфейса.

## 4. UX-принципы
- Отдельные функции, без принудительного сценария.
- Для каждой операции отображаются:
- что читается;
- что будет изменено;
- куда пойдет результат.
- Режимы, меняющие Jira, визуально помечены как рискованные.
- Логи доступны сразу после запуска.

## 5. Конфигурация и параметры запуска
- В `.env` остаются технические параметры (Jira/LLM/авторизация).
- Рабочие параметры прогона (`release parent`, `release target`, issue key) задаются в UI при каждом запуске.

## 6. Логи и отчетность в UI
1. Короткий итог: успех/ошибка, время выполнения.
2. Подробные stdout/stderr.
3. История запусков в интерфейсе.

## 7. Технический подход MVP
1. UI-обертка над существующими CLI-командами.
2. Запуск команд через subprocess c захватом stdout/stderr.
3. Без изменения текущей бизнес-логики генерации.
