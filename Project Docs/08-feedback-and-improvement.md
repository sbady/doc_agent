# Обратная связь и самоулучшение системы

Цель этого документа — чтобы опыт не уходил в стол. Когда владелец (или преемник) на ходу уточняет правило, агент не исполняет его разово, а помогает **зафиксировать** правило в канон, чтобы дальше оно соблюдалось без повторов. Это подход «скилл как обучаемое состояние»: правило проверяется на практике и сохраняется, только если оно действительно улучшает результат.

## Что фиксировать, а что нет

**Не каждое замечание становится правилом.** Разовую правку конкретной задачи выполняем и не трогаем общие инструкции. Петля запускается, только когда замечание:

- влияет на **процесс**, и
- по контексту **похоже на повторяющееся** (отразится на последующих задачах), а не разовая просьба.

Если оба условия есть — агент **предлагает** зафиксировать (не делает молча). Если это разовая правка — просто выполняет, без предложения менять правила.

## Протокол

1. **Предложить (без воды).** Агент коротко выносит в чат предложение в формате: «С учётом вашего запроса предлагаю зафиксировать для задач вида X правило: … Зафиксировать?». Без длинных обоснований, по сути. Не выдумывает — если непонятно, спрашивает.
2. **Дождаться явного подтверждения.** **Без явного «да» от владельца правило в канон/скиллы/`AGENTS.md` не пишется и на будущие задачи не распространяется.** Никаких «принял к сведению и применил ко всему» по умолчанию.
3. **Зафиксировать.** После подтверждения агент дописывает правило в нужный файл и добавляет строку в `Project Docs/CHANGELOG.md` (дата — правило — почему — какой файл изменён).

## Куда какое правило писать

| Тип правила | Файл |
|---|---|
| Процесс по статьям/релизам, правила содержания, формат Jira-комментария, сортировка таблицы | `docs/AGENT_WORKFLOWS.md` (канон) |
| Модель веток, Definition of Done, приоритет источников | `AGENTS.md` (корень) |
| Процедура работы скилла (git, написание статей) | `skills/<name>/SKILL.md` и его `references/` |
| Поведение инструмента `doc_agent` (режимы, ручки) | `docs/` |
| Контекст проекта, продуктовые факты | `Project Docs/01-project-context.md` |

Если правило противоречит существующему — не плодить дубль: править источник на месте и отметить в CHANGELOG, что заменено.

## Принципы

- **Канон — один.** При расхождении побеждает `AGENT_WORKFLOWS.md`; скиллы подчинены ему. Не оставляй два правила, говорящих разное.
- **Только то, что переиспользуется.** Разовая деталь конкретной задачи — не правило; в канон не пишем.
- **Детерминируемое — в скрипт.** Если операция повторяется и её можно автоматизировать (ручка `doc_agent`, скрипт) — лучше довести до инструмента, чем держать в правилах для агента.
- **Согласование обязательно** для правок канона, `AGENTS.md` и скиллов. Агент предлагает — владелец принимает.

## Связанные роли

- Главный ассистент ловит уточнение и ведёт петлю.
- `doc-writer` фиксирует правила содержания/стиля.
- `git-workflow` фиксирует правила веток/публикации.
