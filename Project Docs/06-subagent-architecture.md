# Архитектура Субагентов Для Документации

## Цель

Собрать удобную схему работы с субагентами так, чтобы они:

- писали новые статьи и обновляли существующие;
- учитывали специфику платформы и двух продуктовых контуров;
- соблюдали стиль, структуру и форматирование документации;
- безопасно выполняли Git-операции в рамках принятого workflow.

## Рекомендация По Архитектуре

Оптимальная базовая схема: **2 субагента**, а не один.

### 1. Doc Writer

Зона ответственности:

- анализ Jira-задачи, комментариев, артефактов и вложений;
- поиск затрагиваемой статьи или выбор места для новой статьи;
- написание новой статьи;
- обновление существующей статьи;
- соблюдение синтаксиса Diplodoc/YFM;
- учет различий между открытым и закрытым контурами;
- обновление `toc` при необходимости;
- подготовка короткого change summary для dev-review.

Когда использовать:

- новая статья;
- доработка существующей статьи;
- реструктуризация статьи;
- подготовка черновика по релизной задаче.

### 2. Git Workflow

Зона ответственности:

- создать ветку от `main` по номеру задачи;
- проверить рабочее дерево перед началом;
- после изменений проверить diff и список файлов;
- проверить возможные конфликты относительно целевой ветки;
- подготовить коммит;
- запушить ветку;
- отчитаться, что готово для MR или где есть риск.

Когда использовать:

- старт новой задачи;
- подготовка изменений к push;
- разбор merge/rebase-конфликтов;
- техническая синхронизация веток.

## Почему Лучше Не Объединять Это В Одного Агента

Один агент тоже возможен, но на практике это хуже по двум причинам:

1. Смешиваются две разные ответственности: содержательная редактура и технический Git-workflow.
2. Возрастает риск, что агент с хорошим доступом к Git начнет делать лишние действия, когда задача была только на текст.

Поэтому лучше:

- **Doc Writer** держать как продуктово-редакторского агента;
- **Git Workflow** держать как технического исполнителя;
- главный ассистент использовать как оркестратор между ними.

## Что Уже Зафиксировано В Проектном Контексте

В текущей базе проекта уже есть полезный минимум:

### Из `Project Docs`

- назначение проекта документации;
- структура рабочей папки: `doc_diplo` и `doc_agent`;
- факт существования двух продуктовых версий документации;
- процесс актуализации релизной документации;
- workflow создания и изменения статей;
- логика ревью, dev/prod-публикации;
- базовый формат совместной работы с ассистентом.

### Из `doc_agent`

- уже есть промпт для редактирования статьи по Jira:
  - `doc_agent/prompt_templates/article_update_from_jira.txt`
- в нем уже зафиксированы важные принципы:
  - роль техписа;
  - фокус на пользовательской документации;
  - ориентация на организаторов мероприятий;
  - запрет на выдумывание неподтвержденных деталей;
  - обязательность блока `Требует уточнения`, если данных не хватает.

## Какого Контекста Уже Хватает Для Старта

Чтобы собрать первую рабочую версию Doc Writer, уже достаточно следующего:

- для какого продукта ведется документация;
- что есть два контура;
- что основной репозиторий статей — `doc_diplo`;
- что формат — Diplodoc / YFM;
- что целевая аудитория — в основном организаторы мероприятий;
- что стиль должен быть прикладным, понятным и без лишнего техжаргона.

Этого уже хватает, чтобы сделать первую полезную версию агента.

## Какого Контекста Не Хватает Для Качественной Работы

Чтобы агент работал стабильно и в нужном тебе стиле, нужно еще зафиксировать следующее.

### Для Doc Writer

1. Примеры хороших эталонных статей.
2. Шаблон структуры статьи по типам:
   - новая инструкция;
   - обновление существующей статьи;
   - короткая служебная статья;
   - статья с ветвлением по контурам.
3. Синтаксические правила Diplodoc/YFM:
   - какие блоки допустимы;
   - как оформлять note/warning;
   - как вставлять изображения;
   - как оформлять вкладки, условия, include-блоки.
4. Правила именования файлов и изображений.
5. Правила обновления `toc.yaml`.
6. Правила по терминологии:
   - какие термины использовать;
   - какие нельзя;
   - какие слова считаются внутренними и не должны попадать в пользовательский текст.
7. Правила по различиям WCE/MS:
   - как решать, нужен ли общий текст;
   - когда делать условный блок;
   - когда делать отдельный скриншот.
8. Требования к итоговому результату:
   - нужен готовый файл;
   - нужен только измененный блок;
   - нужен change summary;
   - нужен список открытых вопросов.

### Для Git Workflow

1. Точное правило именования веток.
2. От какой ветки создается рабочая ветка:
   - всегда от `main`;
   - исключения.
3. Формат commit message.
4. Нужно ли агенту делать commit и push автоматически или только по подтверждению.
5. Какой target branch для MR по типам задач.
6. Как проверять конфликты:
   - относительно `DEV`;
   - относительно `main`;
   - относительно обеих веток.
7. Что агенту запрещено делать:
   - например, не выполнять merge в защищенные ветки;
   - не пушить без отдельного указания;
   - не делать `reset --hard`.

## Что Лучше Дать В Качестве Эталонов

Минимальный хороший набор:

1. 5-10 хороших статей из `doc_diplo`.
2. 2-3 статьи с условными блоками по разным контурам.
3. 2-3 примера, где есть изображения и корректное оформление путей.
4. 1 пример новой статьи.
5. 1 пример обновления существующей статьи.
6. 2-3 примера коммитов и веток, которые считаются правильными.

## Входной Контракт Для Doc Writer

На одну задачу лучше подавать такой минимум:

- `task_key`: номер задачи;
- `task_type`: новая статья / обновление статьи;
- `source_context`: описание задачи, комментарии, вложения, скриншоты;
- `target_article`: путь к статье, если известен;
- `target_contour`: `WCE`, `MS` или оба;
- `expected_output`: черновик / готовый файл / измененный блок;
- `constraints`: что точно нельзя менять;
- `done_criteria`: что считается завершением.

Удобный шаблон постановки:

```text
Задача: MSP-XXXX
Тип: новая статья | обновление статьи
Контур: WCE | MS | оба
Путь к статье: ...
Что изменилось: ...
Источники контекста: Jira / комментарии / PDF / скриншоты / устное описание
Что нужно на выходе: готовый md-файл | только измененный блок | черновик
Доп. ограничения: ...
```

## Входной Контракт Для Git Workflow

Удобный шаблон:

```text
Задача: MSP-XXXX
Репозиторий: doc_diplo
Базовая ветка: main
Нужно:
- создать ветку;
- проверить рабочее дерево;
- после изменений показать diff;
- проверить конфликты с DEV;
- сделать commit;
- push на origin
Ограничения:
- без merge в защищенные ветки;
- без push без подтверждения
```

## Рекомендуемый Рабочий Сценарий

### Сценарий 1. Новая статья

1. Git Workflow создает ветку от `main`.
2. Doc Writer анализирует задачу и артефакты.
3. Doc Writer пишет статью и при необходимости обновляет `toc`.
4. Git Workflow проверяет diff и потенциальные конфликты.
5. Ты смотришь результат.
6. Git Workflow делает commit и push.

### Сценарий 2. Изменение существующей статьи

1. Git Workflow создает ветку.
2. Doc Writer читает текущую статью и контекст задачи.
3. Doc Writer вносит изменения точечно.
4. Git Workflow проверяет, какие файлы изменились.
5. При необходимости агент сообщает о рисках по конфликтам.
6. После проверки — commit и push.

### Сценарий 3. Конфликтная ситуация

1. Git Workflow собирает конфликтующие файлы и объясняет, в чем конфликт.
2. Doc Writer подключается только если конфликт затрагивает текст статьи или `toc`.
3. Git Workflow применяет выбранное решение и показывает итоговый diff.

## Практическая Рекомендация По Внедрению

Лучше внедрять не все сразу, а в два шага:

### Этап 1

Собрать **Doc Writer** как основного субагента.

Почему:

- именно он снимает с тебя основную когнитивную нагрузку;
- Git-процесс у тебя уже понятен и относительно стабилен;
- по статьям больше требований к контексту и стилю.

### Этап 2

Добавить **Git Workflow** как вспомогательного субагента.

Почему:

- его проще формализовать;
- он может работать по четкому чек-листу;
- он хорошо дополняет Doc Writer, но не заменяет его.

## Что Я Рекомендую Сделать Следующим

1. Зафиксировать отдельный system prompt для **Doc Writer**.
2. Зафиксировать отдельный system prompt для **Git Workflow**.
3. Собрать короткий `style guide` по статьям.
4. Собрать `git guide` по веткам, commit и push.
5. Выбрать 5-10 эталонных статей и положить их в отдельную папку примеров.

## Короткий Вывод

Для твоего процесса лучше всего подходит схема:

- **1 главный ассистент-оркестратор**
- **1 субагент Doc Writer**
- **1 субагент Git Workflow**

Это даст:

- меньше ручной рутины;
- более предсказуемые правки статей;
- меньше ошибок в Git;
- понятный входной контракт для каждой задачи.
