# Контекст проекта документации

## Назначение

Проект ведется для пользовательской документации платформы проведения мероприятий.

## Репозитории

Работа идёт с двумя отдельными репозиториями, лежащими рядом:

- `doc_agent` (этот репозиторий) — правила, регламенты, скиллы и инструменты автоматизации (релизные заметки, Doc QA, сканер стоп-слов). Хостинг: GitHub.
- `doc_diplo` — исходный код документации на движке Diplodoc. Хостинг: **GitLab**. Клонируется рядом как `../doc_diplo`.

## Продуктовые версии документации

Документация ведётся для **двух очень похожих продуктов** с минимальными различиями, в одном наборе исходников:

1. **MS** (домен `MS`) — открытый контур для внешних заказчиков. Есть функционал **платных мероприятий**: продажа билетов, заказы, оплата.
2. **WCE** (домен `WCE`) — закрытый контур внутри организации. Платной функциональности **нет**.

Различия отражаются условными блоками: `{% if domen == 'WCE' %} ... {% else %} ... {% endif %}`. Контент одного контура не должен «протекать» в другой. В частности, упоминания платности (билеты, заказы, покупка, скидки, промокоды) не должны попадать в WCE. Для контроля этого есть сканер стоп-слов (`stop_words_checker-main/`, см. `04-doc-agent-reference.md`).

## Основные артефакты процесса

- Релизная таблица в Jira (задача актуализации документации релиза).
- Релизные заметки для публикации в разделе `Релизы`.
- Задачи на создание/изменение статей.
- Изменения в статьях `doc_diplo` с публикацией на dev/prod стенды.

## Рабочая цель

Поддерживать документацию в актуальном состоянии, быстро отражать изменения продукта в релизах и статьях, и держать процесс прозрачным для согласующих.
