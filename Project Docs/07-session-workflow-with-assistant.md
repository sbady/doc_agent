# Рабочий Workflow В Этой Сессии

## Цель

Зафиксировать краткий контекст нашего взаимодействия по задачам на создание и изменение статей.

## Базовая Модель Работы

1. Пользователь ставит задачу в чат.
2. По задаче при необходимости создается папка в `Tasks`.
3. Контекст задачи фиксируется в `Tasks/.../Задача и контекст.md`.
4. Если задача требует изменений в `doc_diplo`, работа идет в именной ветке от `main`.
5. Ветка по умолчанию называется как номер задачи, например `MSP-7598`.
6. Содержательная работа по статье выполняется через профиль `doc-writer`.
7. Git-операции выполняются через профиль `git-workflow`.
8. Главный ассистент выступает как оркестратор между этими шагами.

## Что Важно Для Git

- `Tasks` не должны попадать в Git
- в Git должны попадать только чистые изменения статей, `toc.yaml` и связанных ассетов
- нельзя работать напрямую в `main`
- commit и push выполняются только когда это действительно нужно по задаче

## Как Лучше Передавать Задачу

Допустимы оба варианта:

- сразу контекст в чат
- контекст в чат с автоматическим созданием папки задачи в `Tasks`

Рабочий вариант по умолчанию:

- пользователь пишет задачу в чат
- ассистент сам создает папку задачи и фиксирует там контекст

## Роли

### Главный ассистент

- понимает задачу целиком
- подготавливает порядок действий
- решает, когда нужен `doc-writer`
- решает, когда нужен `git-workflow`

### Doc Writer

- пишет новую статью
- обновляет существующую статью
- учитывает стиль, YFM/Diplodoc, контуры и структуру документации

### Git Workflow

- создает и проверяет рабочую ветку
- следит, чтобы `Tasks` не попадали в Git
- проверяет diff, commit, push и риски конфликтов

## Практический Вывод

В рамках этой сессии оптимальная схема такая:

- пользователь работает через чат с главным ассистентом
- главный ассистент оркестрирует `doc-writer` и `git-workflow`
- папка `Tasks` используется как рабочий контекст задачи, но не как часть Git-изменений
