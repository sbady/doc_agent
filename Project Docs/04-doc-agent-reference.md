# Doc Agent: краткая справка

`doc_agent` — инструмент для ускорения актуализации документации и Doc QA.

## Основные возможности

- Заполнение релизной таблицы в Jira по задачам релиза (`mode fill`).
- Просмотр релизной таблицы без записи (`mode view`, `mode fill-preview`).
- Генерация краткой релизной строки по одной задаче (`--issue-short`).
- Compare эталона и сгенерированного текста с LLM-оценкой (`mode compare`).
- Генерация готового changelog-текста для вставки в исходники документации (`mode changelog-preview`).
- Улучшение формулировок в таблице (`mode refine-preview` / `mode refine-apply`).
- Проверка санитайзера и проверка документации на стоп-слова.

## Базовые входные данные

- Jira: `JIRA_BASE_URL`, `JIRA_AUTH_TYPE`, `JIRA_API_TOKEN` (+ `JIRA_EMAIL` для basic).
- LLM: `LLM_ENDPOINT`, `LLM_MODEL` (и при необходимости `LLM_API_KEY`).
- Для релизных режимов: `RELEASE_PARENT_KEY`, `TARGET_ISSUE_KEY` (или передача через CLI/UI).

## Важные ограничения

- Режимы записи в Jira рассчитаны на wiki-формат описания (Jira Server/DC).
- Для Jira Cloud (ADF) нужна отдельная реализация обновления.
- При обновлении релизной таблицы инструмент сохраняет пользовательские колонки `Шаблоны` и `Мануал`.

## Сканер стоп-слов (BAN-слова)

`stop_words_checker-main/` — инструмент, который обходит документацию и ищет запрещённые термины по словарю `stop_words_checker-main/stopwords.txt`. Назначение — следить, чтобы в **WCE**-контур не попала функциональность и формулировки, которых там быть не должно (платность: билеты, заказы, покупка, продажа, скидки, промокоды, выставление счёта и т.п.).

Как работает:
- обходит страницы документации (стартовый `URL` из `.env`), извлекает текст и ищет совпадения;
- по умолчанию регистронезависимо; обычные строки ищутся как целое слово/фраза, строки с `re:` — как regex;
- `allow:` задаёт разрешённый контекст для предыдущего стоп-слова (например, `рейтинг` запрещён, но `опрос-рейтинг` допустим);
- выводит отчёт в консоль с контекстом каждого совпадения, опционально сохраняет JSON.

Запуск (из `stop_words_checker-main/`):

```bash
# одна страница (debug)
python debug/check_single_page.py --path documentation/first_steps/reg.html --stop-words stopwords.txt
# полный обход
python stopword_crawler.py --max-pages 50 --output report.json
```

Стартовый URL и путь к словарю берутся из `.env` (`URL`, `STOP_WORDS_FILE`), либо переопределяются флагами `--url` / `--stop-words`. Подробности и формат словаря — в `stop_words_checker-main/README.md`.

**Когда запускать (правило).** Проверку нужно прогонять регулярно и **обязательно** — если в правках были упоминания платной функциональности или других терминов из словаря стоп-слов (затронуты общие или WCE-статьи про мероприятия, заказы, билеты, оплату). Это часть Definition of Done (см. `AGENTS.md`).

## Связь с основным процессом документации

`doc_agent` используется на этапе формирования релизной таблицы и релизных заметок, для ускорения подготовки задач на изменения статей, а также для Doc QA (проверка на стоп-слова).
