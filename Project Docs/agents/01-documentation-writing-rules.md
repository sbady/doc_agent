# Правила написания документации

## Назначение

Этот файл задает единые правила для написания и редактирования статей в `doc_diplo`.
Его нужно использовать как базовую инструкцию для агентов и ассистентов, которые работают с пользовательской документацией проекта.

Цель:
* сделать статьи однообразными
* уменьшить стилистический шум
* снизить количество несогласованных формулировок и форматирования

## Общий принцип

Документация должна быть:
* пользовательской
* спокойной по тону
* понятной без внутреннего контекста команды
* согласованной по структуре и оформлению

Статья должна объяснять, как работает интерфейс и что делает пользователь, а не пересказывать внутреннюю реализацию.

## Стиль текста

Используйте:
* нейтральный деловой тон
* простые формулировки
* настоящее время
* пользовательскую терминологию продукта

Избегайте:
* избыточно технических формулировок
* внутренних терминов команды, если они не нужны пользователю
* тяжеловесных и канцелярских оборотов

## Выделение текста

Основное правило:
* не перегружать статьи жирным выделением

Используйте:
* курсив для мягкого акцента
* жирный текст только там, где действительно нужно выделить важный элемент интерфейса, предупреждение или ключевое ограничение

Нельзя:
* делать жирным большие куски текста
* использовать жирный текст как основной способ оформления статьи
* выделять подряд много элементов без необходимости

Если можно обойтись без выделения, лучше обойтись без него.

## Кавычки

Используйте только прямые кавычки:
* `"..."` для названий кнопок, вкладок, блоков и полей

Не используйте:
* типографские кавычки
* смешение разных видов кавычек в одной статье

## Списки

Для маркированных и нумерованных списков:
* не ставить точку в конце пункта, если это не полное самостоятельное предложение
* не ставить лишние знаки препинания в конце коротких пунктов

Точка нужна:
* только если пункт является полноценным предложением
* только если строка действительно оформлена как предложение, а не как элемент перечня

Примеры:

Правильно:
* Откройте вкладку "Статистика"
* Нажмите кнопку "Сгенерировать"
* Скачайте файл

Тоже правильно:
* Если файл уже был сгенерирован, повторно нажмите кнопку "Скачать".

Неправильно:
* Откройте вкладку "Статистика".
* Нажмите кнопку "Сгенерировать".
* Скачайте файл.

## Структура статьи

Структура статьи должна повторять структуру интерфейса.

Предпочтительный порядок:
1. где находится раздел или функция
2. какие блоки и элементы есть в интерфейсе
3. как пользователь выполняет действия
4. что происходит в результате
5. ограничения, особенности, примечания

Не нужно:
* строить статью вокруг внутренней логики бэкенда
* выносить теоретические объяснения впереди практического сценария

## Названия элементов интерфейса

Названия кнопок, вкладок, разделов и блоков:
* писать ровно так, как они называются в интерфейсе
* брать в прямые кавычки

Если у функции есть несколько связанных элементов:
* сначала назвать раздел
* потом вкладку
* потом конкретное действие

## Формулировки ограничений и условий

Если логика подтверждена не полностью:
* писать нейтрально
* не обещать пользователю поведение, которое не подтверждено задачей, интерфейсом или артефактами

Если есть сомнение:
* лучше ограничить формулировку
* не додумывать продуктовую логику

## Работа с контурами

Если функционал отличается между `MS` и `WCE`:
* использовать условные блоки
* не смешивать контуры в одном общем описании

Если функционал относится только к одному контуру:
* это должно быть явно видно в статье

Перед правками, где есть контурные различия, сверяйтесь с накопительным списком тем в `doc_agent/skills/doc-writer/references/contour-specific-topics.md`. Портальные темы относятся только к `WCE` и не должны попадать в текст для `MS`; платные мероприятия относятся только к `MS` и не должны попадать в текст для `WCE`.

## Когда обновляется существующая статья

При редактировании статьи:
* не вносить лишние stylistic cleanup-правки далеко за пределами задачи
* но в затронутом фрагменте приводить оформление к единообразному виду

Допустимо:
* убрать лишние точки в списках рядом с изменяемым блоком
* выровнять кавычки в изменяемом фрагменте
* убрать лишние жирные выделения в изменяемом фрагменте

Не нужно:
* переписывать половину статьи "заодно"
* менять соседние разделы без связи с задачей

## Минимальный чек перед сдачей

Перед завершением работы нужно проверить:
* нет ли лишнего жирного текста
* используются ли прямые кавычки
* нет ли точек в конце коротких пунктов списков
* соответствует ли структура статьи интерфейсу
* нет ли неподтвержденных обещаний о поведении функции
* правильно ли разделены контуры `MS` и `WCE`

## Короткая инструкция для агента

Если агент пишет или редактирует статью, он должен:
1. сохранить пользовательский стиль
2. не злоупотреблять жирным текстом
3. чаще использовать нейтральный текст и точечный курсив
4. использовать только прямые кавычки
5. не ставить точки в конце коротких пунктов списков
6. повторять структуру интерфейса
7. не додумывать продуктовую логику
8. делать только изменения в рамках задачи, если не согласовано иное
