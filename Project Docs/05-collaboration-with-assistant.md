# Работа с ассистентом в проекте

Этот документ фиксирует, как использовать ассистента как постоянного помощника по документации.

## Что можно делегировать ассистенту

- Обновление существующих статей в `doc_diplo` по вашему контексту.
- Подготовка новых статей и черновиков разделов.
- Рефакторинг структуры статьи (логика, заголовки, навигация, единый стиль).
- Подготовка/редактура релизных заметок.
- Проверка консистентности формулировок между открытым и закрытым контурами.
- Помощь с Git-операциями: разбор конфликтов, безопасное разрешение merge/rebase конфликтов.
- Подготовка служебной документации по процессам проекта.

## Как лучше ставить задачу ассистенту

Минимум входных данных:

1. Что меняется в продукте.
2. Какие статьи затрагиваются (если известно).
3. Для какого контура изменения (WCE, MS или оба).
4. Нужен ли только черновик или сразу правка файлов.
5. Нужен ли итог под dev-публикацию (с коротким change summary).

## Формат ожидаемого результата

- Конкретные изменения в файлах.
- Краткий список того, что изменено.
- Отдельный список открытых вопросов/рисков (если есть).
- При необходимости — рекомендации по следующему шагу (проверка, публикация, согласование).

## Правило актуальности контекста

Если процесс, шаблоны Jira или правила публикации изменятся, обновляйте файлы в `Project Docs` в первую очередь, чтобы ассистент всегда опирался на актуальную базу.
