# START HERE — руководство для техписа

Это входная точка для нового сотрудника, который ведёт документацию платформы. Здесь — как всё устроено и как работать с агентами (Codex / Claude Code), чтобы быстро делать релизы и статьи.

## За 5 минут: что это и как устроено

Есть **два отдельных репозитория**:

- **этот репозиторий (`doc_agent`)** — правила, регламенты, скиллы и инструменты автоматизации (релизная таблица, релизные заметки, Doc QA, сканер стоп-слов). Хостинг: GitHub. Это точка входа — его клонируешь первым.
- **`doc_diplo`** — сами статьи (Diplodoc/YFM). Хостинг: **GitLab**. Клонируется рядом как `../doc_diplo`.

Документация ведётся для **двух очень похожих продуктов**:
- **MS** — открытый контур, есть платные мероприятия (продажа билетов, заказы, оплата);
- **WCE** — закрытый контур, платной функциональности нет.

Различия — условными блоками `{% if domen == 'WCE' %} ... {% else %} ... {% endif %}`. Платность не должна «протекать» в WCE — для контроля есть сканер стоп-слов.

Задачи приходят из **Jira** с ключами `MSP-XXXX`.

## Первичная настройка (один раз)

1. Склонировать этот репозиторий (`doc_agent`) и открыть его как рабочую папку.
2. Подключить репозиторий документации: дать агенту ссылку на GitLab-репозиторий `doc_diplo` (или попросить склонировать) — он должен лежать рядом как `../doc_diplo`. Для clone/push/MR нужна авторизация в GitLab; если она уже выполнена — всё работает, иначе попроси подключить доступ.
3. Проверить, что в `doc_diplo` есть и запушены ветки `main` (прод) и `DEV` (дев-стенд).
4. `doc_agent`: создан `.venv`, в `.env` заданы Jira и LLM (`JIRA_BASE_URL`, `JIRA_API_TOKEN`, `JIRA_AUTH_TYPE`, `LLM_ENDPOINT`, `LLM_API_KEY`, `LLM_MODEL`). Образец — `env_example`.

## Три главных сценария

### 1. Заполнить/вычитать релизную таблицу
Нужны два Jira-ключа: `release_parent` (источник задач релиза) и `release_target` (задача актуализации документации). Агент через `doc_agent` заполняет таблицу, сортирует, проставляет `Стенд`, `Тип задачи`, `Краткое описание`, `Мануал`. Команды — в `AGENT_WORKFLOWS.md`. Результат — таблица, готовая к согласованию.

### 2. Создать/изменить статью по задаче
Дай агенту Jira-ключ (`MSP-XXXX`). По умолчанию агент: ветка от `main` по ключу → правки в `doc_diplo` → публикация в `DEV` + ссылка на MR → зеркало в `main` + MR → проверка конфликтов в обе стороны → (если затронута платность/WCE — прогон стоп-слов) → **готовый комментарий для Jira**. Ссылку и комментарий отдельно просить не нужно — это часть результата.

### 3. Перенести согласованные релизные заметки в документацию
Нужны `release_target` и `release_version`. Согласованная таблица — read-only: агент не переформулирует, только переносит в `releases.md`, пушит в обе ветки и отдаёт ссылки на MR.

## Что ты всегда получаешь по задаче на статью

- текущую ветку (= Jira-ключ);
- список изменённых статей с кратким «что изменено»;
- **ссылки на MR в `DEV` и `main`**;
- результат проверки конфликтов;
- результат стоп-слов (если запускался);
- **готовый комментарий для Jira** (для проверяющего, со ссылками на дев-стенд).

Если чего-то нет — агент обязан написать, чего именно и почему.

## Как ставить задачу агенту

Минимум — Jira-ключ. Полезно: что меняется в продукте, какие статьи затронуты, контур (MS/WCE/оба), нужен ли только черновик. Если хочешь только черновик без публикации — скажи явно «только покажи / не пушь».

## Как система улучшается

Если агент сделал не так или ты сформулировал новое правило — попроси зафиксировать его (см. `08-feedback-and-improvement.md`). Агент предложит формулировку, ты согласуешь, он впишет в нужный файл и в `CHANGELOG.md`. Так правила не приходится повторять.
