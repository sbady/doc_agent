# AGENTS.md — Конституция работы с документацией

Это **репозиторий правил, регламентов и инструментов** для ведения пользовательской документации платформы проведения мероприятий. Он — единая точка входа для техписа и для AI-агентов (Codex, Claude Code и др.). Здесь живут процессы, скиллы и инструментарий (`doc_agent`). **Сама документация (статьи) живёт в отдельном репозитории `doc_diplo`** и подтягивается рядом.

**Прочитай этот файл целиком, прежде чем что-либо делать.** Для задач на статьи и релизы дальше обязателен `docs/AGENT_WORKFLOWS.md` (канон процесса).

Контент документации — на русском. Отвечай владельцу по-русски, если не попросили иначе.

## Структура рабочего пространства

Два отдельных репозитория, лежащих рядом (соседние папки):

| Репозиторий | Где | Что | Хостинг |
|---|---|---|---|
| **этот репозиторий** (`doc_agent`) | корень = текущая папка | правила, регламенты, скиллы + инструменты автоматизации | GitHub |
| **`doc_diplo`** | соседняя папка `../doc_diplo` | исходники документации (статьи, Diplodoc/YFM) | **GitLab** |

Соглашение о путях в документах:
- пути без префикса (`docs/`, `.venv`, `.env`, `ui/`, `skills/`, `Project Docs/`, `stop_words_checker-main/`) — относительно корня **этого** репозитория;
- пути вида `doc_diplo/...` — файлы в репозитории документации; локально это `../doc_diplo/...`;
- команды `doc_agent` запускаются из корня этого репозитория (`& ".\.venv\Scripts\python.exe" ".\main.py" ...`);
- `Tasks` — локальная папка с контекстом задач рядом с репозиториями; **никогда не коммитится** ни в один репозиторий.

## Подключение к репозиторию документации (GitLab)

Перед работой со статьями документация (`doc_diplo`) должна быть склонирована рядом как `../doc_diplo`.

1. Если папки `../doc_diplo` нет — попросить у владельца ссылку на GitLab-репозиторий документации и склонировать его соседней папкой (либо попросить указать путь, если он уже есть локально).
2. Работа с GitLab (clone, push, создание merge request) требует доступа. Если авторизация в GitLab уже выполнена (git-креденшелы / `glab` / SSH-ключ) — всё работает. Если нет — остановиться и попросить владельца подключить доступ, не выдумывать обходные пути.
3. В `doc_diplo` должны существовать и быть запушены две долгоживущие ветки: `main` (прод) и `DEV` (дев-стенд).

## Что это за проект

- Цель: держать пользовательскую документацию в актуальном состоянии, быстро отражать изменения продукта в релизах и статьях, держать процесс прозрачным для согласующих.
- Задачи приходят из **Jira** с ключами `MSP-XXXX`.

### Два продукта (два контура)

Документация ведётся для **двух очень похожих продуктов** с минимальными различиями, в одном наборе исходников:

- **MS** (домен `MS`) — открытый контур для внешних заказчиков. Здесь есть функционал **платных мероприятий**: продажа билетов, заказы, оплата.
- **WCE** (домен `WCE`) — закрытый контур внутри организации. Платной функциональности **нет**.

Различия фиксируются **условными блоками** в статьях: `{% if domen == 'WCE' %} ... {% else %} ... {% endif %}`. Контентом, специфичным для одного контура, нельзя «протекать» в другой. Особенно: упоминания платности (билеты, заказы, покупка, скидки, промокоды и т.п.) не должны попадать в WCE-вывод. Для контроля этого есть сканер стоп-слов (см. ниже).

## Приоритет источников (что каноно при расхождении)

1. **`docs/AGENT_WORKFLOWS.md`** — канон для задач на статьи и релизы. При расхождении побеждает он.
2. **Этот `AGENTS.md`** — точка входа, топология, модель веток и Definition of Done по умолчанию.
3. **`Project Docs/`** — контекст и онбординг.
4. **`skills/`** — процедуры. Подчинены канону. Если скилл противоречит канону — прав канон, скилл нужно поправить через петлю обратной связи.

## Ориентация в начале работы

1. Прочитать этот файл.
2. Убедиться, что `../doc_diplo` доступен (см. «Подключение к GitLab»).
3. Для задачи на статью/релиз — прочитать `docs/AGENT_WORKFLOWS.md` до планирования, выбора статей и создания ветки.
4. При нехватке контекста — прочитать Jira-задачу по ключу (описание, комментарии, блок `Документация Diplodoc`, стенды, связанные задачи).
5. Не просить владельца переобъяснять то, что уже есть в этих файлах.

## Модель веток (doc_diplo)

В `doc_diplo` две долгоживущие рабочие ветки: `main` (прод) и `DEV` (дев-стенд для согласования). Обе должны существовать и быть запушены. Если рабочей `main`-ветки нет или она не запушена — это ошибка состояния: создать/запушить, прежде чем делать MR.

- Рабочая ветка под задачу создаётся **от `main`** и называется **точно как Jira-ключ** (`MSP-9479`). От `main` — чтобы не утянуть несогласованные изменения, которые есть только в `DEV`.
- Изменения сначала в **`DEV`** (дев-стенд, согласование), затем зеркально в **`main`** (прод). `DEV` и `main` не должны расходиться по согласованному контенту.
- Если `DEV` и `main` существенно разошлись — **нельзя** вливать `DEV` в ветку для `main`. Отдельная ветка от `DEV` под DEV-MR и отдельная ветка от `main` под main-MR (детали — в `docs/AGENT_WORKFLOWS.md`).
- Прямые правки в `main`/`DEV` запрещены — только через MR.

## Definition of Done — поведение по умолчанию

Если задача подразумевает **внесение правок в статью или релиз** (а не «только черновик» / «только покажи»), агент по умолчанию, **без отдельного запроса**, доводит её до результата:

1. Создаёт ветку от `main` по ключу задачи и вносит правки (через `doc-writer`).
2. Проверяет diff: только нужные файлы статей / `toc.yaml` / ассеты, без `Tasks` и мусора.
3. **Публикует в `DEV`**: commit, push, **создаёт merge request в `DEV` и отдаёт ссылку** — по умолчанию, без переспрашивания.
4. **Зеркалит в `main`**: те же изменения в рабочую `main`-ветку, push, MR в `main`, отдаёт ссылку.
5. Проверяет конфликты против **обеих** целевых веток отдельно.
6. **Если правки затронули платную функциональность или общие/WCE-статьи — прогоняет сканер стоп-слов** (см. ниже) и сообщает результат.
7. Готовит **комментарий для Jira** (формат — в `docs/AGENT_WORKFLOWS.md`): для проверяющего, со ссылками на **дев-стенд**, без MR/коммитов/веток внутри блока.

В финальном ответе владелец **всегда** получает: ветку, изменённые статьи, **ссылки на MR в `DEV` и `main`**, результат проверки конфликтов, результат стоп-слов (если запускался) и **готовый Jira-комментарий**. Если шаг не сделан — явно написать, чего нет и почему.

DoD **не** применяется (только подготовка и показ, без commit/push/MR), когда: владелец сказал «только черновик» / «не пушь»; задача — заполнение/вычитка релизной таблицы в Jira; не хватает контекста для безопасной правки (тогда остановиться и спросить).

## Контроль стоп-слов (BAN-слова)

Инструмент `stop_words_checker-main/` обходит документацию и ищет запрещённые термины по словарю `stop_words_checker-main/stopwords.txt` (платность, заказы, покупка, продажа, промокоды, скидки и т.п.). Назначение — следить, чтобы в **WCE**-контур не попала функциональность и формулировки, которых там быть не должно.

Правило: **прогонять проверку регулярно**, и обязательно — если в изменениях были упоминания платной функциональности или других терминов из словаря стоп-слов (например затронуты общие или WCE-статьи про мероприятия, заказы, билеты). Запуск и формат словаря — в `Project Docs/04-doc-agent-reference.md` и `stop_words_checker-main/README.md`.

## Безопасность git

- Стейджить **только** изменения статей `doc_diplo`, `toc.yaml` при изменении навигации и связанные ассеты. Никогда — `Tasks`, локальные черновики, конфиги, `.env`.
- Никаких `reset --hard`, force-push, переписывания истории без явной просьбы.
- Коммитить и пушить **только свои** изменения по явным путям. Не подметать чужие незакоммиченные правки.
- Конфликты (часто бинарные скриншоты, `toc.yaml`): определить файлы, объяснить тип, разрешить явно, ничего не потеряв; при семантике статьи — подключить `doc-writer`.

## Петля обратной связи и самоулучшение

Опыт не должен теряться. Когда владелец уточняет или поправляет правило, агент предлагает **зафиксировать** его в канон — после согласования. Протокол — в `Project Docs/08-feedback-and-improvement.md`. Коротко: сформулировать правило → показать владельцу → после «ок» вписать в нужный файл и в `Project Docs/CHANGELOG.md`. Так система дорабатывается сама, а правила не приходится повторять.
